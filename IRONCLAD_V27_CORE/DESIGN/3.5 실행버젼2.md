대표님이 올린 **두 설계도**를 비교하면 핵심은 이것입니다.

* 3.5 설계도 → **scheduler 제거, 단순 엔진**
* 3.1 설계도 → **scheduler + 상태기계 구조**

실전 자동매매 안정성 기준에서는 **3.1 구조가 더 안전합니다.**
이유는 **세션 관리 + 상태기계 + FORCE_EXIT 청산 보장**이 있기 때문입니다. 

그래서 아래에 **충돌 없이 정리된 “완성형 설계도”**를 드립니다.
(대표님 철학: SSOT / 단일 파이프라인 / SAFE_HALT 유지)

# ③ run.py 상태기계 (핵심 안정 구조)

```
PRE_OPEN
   ↓
TRADING
   ↓
NO_NEW_ENTRY
   ↓
FORCE_EXIT
   ↓
CLOSE
```

### 상태 의미

PRE_OPEN
→ 거래 시작 전

TRADING
→ 정상 거래

NO_NEW_ENTRY
→ 신규 진입 금지

FORCE_EXIT
→ 남은 포지션 강제 청산

CLOSE
→ 거래 종료

# ④ FORCE_EXIT 안전 시퀀스

만약 청산 실패하면

```
1 Cancel All Open Orders
2 거래소 상태 재조회
3 incident 기록
4 SAFE_HALT
```

이 단계는 **run.py만 수행**합니다.

# ⑨ 안정성 핵심 6가지

대표님 시스템의 핵심 안정 구조

```
SSOT
schema
stage_contract
integrity_guard
state_machine
FORCE_EXIT
```

이 6개가 있으면

**개인 자동매매 시스템 기준 최상위 안정 구조**입니다.

```
구조 설계        완료

```
